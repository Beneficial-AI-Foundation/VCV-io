# Paper-to-Lean Correspondence

Maps definitions and notation from Alwen-Coretti-Dodis (2020)
"The Double Ratchet: Security Notions, Proofs, and Modularization
for the Signal Protocol" to the Lean 4 formalization in this project.

## Why `Module F G` Instead of Just ℤ/pℤ?

A cyclic group of prime order p IS ℤ/pℤ, canonically. The whole
`Module F G` + `hg : Bijective` dance is just saying "G ≅ F via a ↦ a • g"
with extra steps.

The reason VCV-io uses this API: they want to keep F (scalars/exponents) and
G (group elements) as **separate types** so you can instantiate G with an
elliptic curve group where the representations differ (Curve25519 points vs.
ZMod p scalars). The bijectivity hypothesis is the tax on that abstraction.

For pure math, you could just write DDH directly in ZMod p: "distinguish
(a, b, a\*b) from (a, b, c) for uniform a, b, c in ZMod p." No Module, no
bijectivity, no generator. That's cleaner and equivalent.

This is a VCV-io API design choice, not a mathematical necessity. See
`doc/vcv-io-dependency.md` for a detailed analysis of whether to keep this
abstraction or simplify, and what a ZMod p-based formulation would look like.

## Where is Epsilon?

The paper states Theorem 3 as:

> If G is (t, ε)-DDH-secure, then DDH-CKA is (t, Δ=1, ε)-secure.

This means: if every t-time adversary has DDH advantage ≤ ε, then every
t-time adversary has CKA advantage ≤ ε (same epsilon, tight reduction).

In Lean, we state this **per-adversary** rather than quantifying over epsilon:

```lean
theorem ddh_implies_cka_security (g : G)
    (hg : Function.Bijective (· • g : F → G))
    (adversary : CKAAdversary G G) :
    ckaDistAdvantage (ddhCKA (F := F) g) adversary ≤
      ddhDistAdvantage (F := F) g (ckaAdvToDDHAdv adversary)
```

This is strictly more informative than the paper's statement: for ANY adversary A,
the CKA advantage of A is bounded by the DDH advantage of the *specific* reduced
adversary B = ckaAdvToDDHAdv(A). The paper's epsilon-form follows by taking the
supremum over all adversaries on both sides.

Equivalently: if you fix `ε` and assume `∀ B, ddhDistAdvantage g B ≤ ε`, then
instantiating with `B = ckaAdvToDDHAdv A` gives `ckaDistAdvantage ... A ≤ ε`.

This is the **single-epoch warmup surface**. The paper-faithful Figure 3 theorem
surface is `ddh_implies_figure3_cka_security`; its executable reduction lives in
`Theorems/Reduction.lean`.

## Two-State Design

The paper uses a single state type γ for both sender and receiver. The Lean
formalization splits this into `SenderState` and `ReceiverState` that swap
after each operation:

- After `send`: party holds `ReceiverState` (will next receive)
- After `recv`: party holds `SenderState` (will next send)

For DDH-CKA this is natural:
- `SenderState = G` (peer's public key h)
- `ReceiverState = F` (own secret scalar x)

The paper's single γ implicitly alternates between these types via protocol
convention. Our formalization enforces this alternation at the type level.

## Security Game: Three Levels

The formalization provides three levels of game formalization:

### Level 1: Single-epoch game (`CKA/SecurityGame.lean`)

- `ckaRealExp`: sample key, init, send once, give adversary `(msg, real_output)`
- `ckaRandExp`: same but replace output with uniform random
- `ckaDistAdvantage`: `|Pr[real] - Pr[rand]|`
- `CKASecure`: `∀ adversary, ckaDistAdvantage ≤ ε`

Used for the concrete per-adversary reduction (`ddh_implies_cka_security`).

### Level 2: Figure 3 oracle game (`CKA/Figure3Game.lean`) — paper-faithful

The full adaptive oracle game from the paper, parameterized by Δ:

- `Figure3Adversary`: `OracleComp (unifSpec + ckaOracleSpec) Bool` — adaptive
  adversary with oracle access to `send-P`, `send-P'(r)`, `receive-P`,
  `chall-P`, `corr-P`
- `CKAGameState`: tracks both party states (`SenderState ⊕ ReceiverState`),
  per-party epoch counters, ping-pong phase, pending message, challenge status
- `ckaGameQueryImpl`: stateful oracle implementation in `StateT CKAGameState ProbComp`,
  following the VCV-io PRF/PRFTagReader pattern
- `allowCorrFig3` / `finishedParty` / `corruptionPermittedFig3`: party-specific
  corruption predicates using `max(epochA, epochB)`
- `GamePhase`: explicit ping-pong turn discipline (`awaitingSend` / `awaitingReceive`)
- `figure3Advantage`: `|Pr[A=1 | real] - Pr[A=1 | random]|`
- `Figure3CKASecure cka delta ε`: `∀ tStar adversary, figure3Advantage ≤ ε`

Used for the paper-faithful theorem (`ddh_implies_figure3_cka_security` with Δ=1).
The Figure 3 DDH reduction itself lives in `Theorems/Reduction.lean`.

### Level 3: Restricted multi-epoch game (`CKA/MultiEpochGame.lean`) — auxiliary

A non-adaptive transcript-level approximation, kept as an auxiliary:

- `MultiEpochCKAAdversary`: commits to `tStar`, `numEpochs`, corruption
  requests upfront, then sees the full transcript before guessing
- `CKASecureDelta cka delta ε`: `∀ adversary, multiEpochAdvantage ≤ ε`

Not the paper's full Figure 3 model — see Level 2 for that.

### Bridge

`CKASecureDelta_implies_CKASecure`: any single-epoch adversary lifts to a
1-epoch multi-epoch adversary, so `CKASecureDelta` implies `CKASecure`.
